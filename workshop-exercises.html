A static, step-by-step workshop guide tailored for Altimetrik teams using OpenAI Codex. Complete each exercise in sequence, mark progress, and use App or CLI based on your preference. The core arc is setup, context, planning, implementation, verification, and review.

Why Codex Here

  • Trace issues across frontend, backend, and tests in one loop.
  • Generate constrained diffs with minimal blast radius.
  • Use repo instructions, skills, and browser tools intentionally.
  • Close the loop with test verification plus browser and visual QA.

Client Outcomes

  • Exercises 1-2: establish Codex context and debug contract drift safely.
  • Exercises 3-6: ship scoped fixes and features with artifacts and tests.
  • Exercises 7-8: validate with skills, browser automation, and screenshots.
  • Exercises 9-10: optional follow-ons for expansion after the core path.

What Codex Teaches

  • Use AGENTS.md and repo artifacts to keep context durable.
  • Plan before implementing multi-file work.
  • Choose the right verification surface for the problem.
  • Package changes for review without touching main.

Exercise 1: Install Codex (App or CLI) and Connect Project

10-20 minutes Workshop Setup

Goal: connect Codex to the repo, create a review branch, establish repo context, and produce an architecture artifact.

Expected outputs

  • A review branch separate from main.
  • An updated output/architecture-overview.md.
  • A clear explanation of how the app starts and how the baseline is intentionally seeded.

Choose your path

  1. Open Codex App and sign in with ChatGPT.
  2. Select this repository folder.
  3. Create a review branch in your terminal: 
    git checkout -b codex/<your-workshop-branch>
  4. Run these orientation prompts:
    • Read AGENTS.md and summarize the project rules I should follow.
    • Create an architecture overview of this repository and save it to output/architecture-overview.md.
    • Tell me which tests are intentionally failing on the workshop baseline and why.
    • Start the backend and frontend locally and tell me how to verify they are running.
    • Map dashboard summary data from backend to frontend.
  1. Install CLI: 
    npm install -g @openai/codex
# or
brew install --cask codex
  2. Start Codex inside repo: 
    codex
  3. Create a review branch: 
    git checkout -b codex/<your-workshop-branch>
  4. Sign in and run the same orientation prompts as App path.

Acceptance criteria

  • Codex (App or CLI) can read local files accurately.
  • You can run prompts and get architecture-accurate answers.
  • A review branch exists and output/architecture-overview.md has been updated.

Exercise 3: Incident Response - Low Stock KPI Mismatch

10-20 minutes Incident Response

Goal: triage and resolve a production-style KPI mismatch affecting planning decisions.

Codex setup (App or CLI)

  • Codex App: Reuse your repo session in Codex App and run: Treat this as incident response: validate low-stock KPI mismatch evidence, apply minimal fix for reorder-point equality, rerun regressions, and summarize RCA.
  • Codex CLI: Run codex in repo root (or reuse your open session), then run the same prompt.

Incident brief

  • Planning report understated low-stock exposure for procurement review.
  • Open incident packet: docs/incidents/INC-2026-03-low-stock-kpi.md.
  • Working hypothesis: equality at reorder point is excluded from KPI count.

Run regression checks

source .venv/bin/activate
cd tests
pytest backend/test_bug_regressions.py -q

Tasks

  1. Confirm symptom and impacted KPI from incident packet and tests.
  2. Use Codex to locate low-stock calculation in backend summary endpoint.
  3. Patch the conditional to include equality case.
  4. Re-run the regression file.
  5. Add a short RCA note to output/incident-rca.md (3-5 bullets).

Acceptance criteria

  • Items at reorder point are counted as low stock.
  • Backend regression for this KPI incident passes.
  • output/incident-rca.md documents root cause and mitigation.

Exercise 5: Feature A - Supplier Lead Time Risk API + UI

10-20 minutes Planned Build

Goal: implement risk endpoint and render supplier risk table in dashboard.

Plan-first artifact

  • Create or update output/spec.md before implementation.
  • Capture the problem, acceptance criteria, backend/frontend changes, and verification plan.

Codex setup (App or CLI)

  • Codex App: Reuse your repo session in Codex App and run: Before coding, draft a concise spec in output/spec.md for Feature A. Then implement GET /api/risk/suppliers with warehouse/category filters, render supplier risk rows in dashboard, and update tests.
  • Codex CLI: Run codex in repo root (or reuse your open session), then run the same prompt.

Contract

Run feature tests

source .venv/bin/activate

pytest backend/test_feature_endpoints.py -q

Acceptance criteria

  • UI table renders and updates from API.
  • output/spec.md reflects the accepted implementation plan.

Exercise 7: Playwright CLI Skill Feature - Add Orders Search UX

10-20 minutes Playwright Skill

Goal: add an Order Search input and validate real browser behavior with Playwright CLI skill flow.

Codex setup (App or CLI)

  • Codex App: Reuse your repo session in Codex App and run: Add Order Search UX in orders view (id, customer, status) and validate behavior with Playwright CLI snapshots and screenshots.
  • Codex CLI: Run codex in repo root (or reuse your open session), then run the same prompt.

Feature to add

  • Add a search field in Orders list filtering by order id, customer, and status.
  • Filtering should update visible orders in real time.

Why a skill here

  • This step turns browser QA into a reusable workflow instead of one-off manual checking.
  • Read the repo sample skill if you want a local example of how workflows can be packaged.

Playwright CLI skill setup

command -v npx >/dev/null 2>&1
export CODEX_HOME="${CODEX_HOME:-$HOME/.codex}"
export PWCLI="$CODEX_HOME/skills/playwright/scripts/playwright_cli.sh"

Validation flow

$PWCLI open http://localhost:3000 --headed
$PWCLI snapshot
# use current snapshot refs
$PWCLI type eX ORD-5002
$PWCLI snapshot
$PWCLI screenshot

Acceptance criteria

  • Search input exists and filters list correctly.
  • Clearing search restores full list.
  • Playwright flow verifies behavior with no regressions.
  • Browser evidence is saved or summarized for review.

Exercise 8: Screenshot Capture Skill - Final Visual QA

10-20 minutes Screenshot Skill

Goal: validate visually that all built features are present and correct using Screenshot Capture.

Codex setup (App or CLI)

  • Codex App: Reuse your repo session in Codex App and run: Run final visual QA with screenshot capture. Produce captures for KPI correctness, supplier risk table, order search, and export controls.
  • Codex CLI: Run codex in repo root (or reuse your open session), then run the same prompt.

Skill setup

export CODEX_HOME="${CODEX_HOME:-$HOME/.codex}"
export SS_SKILL="$CODEX_HOME/skills/screenshot"

macOS permission preflight

bash "$SS_SKILL/scripts/ensure_macos_permissions.sh"

Capture checklist

  1. Dashboard with corrected KPI values visible.
  2. Supplier risk table populated and readable.
  3. Orders view with search filtering active.
  4. Export CSV control visible in Orders view.
  5. Optional reviewer summary drafted from docs/templates/reviewer-summary-template.md.

Capture commands

python3 "$SS_SKILL/scripts/take_screenshot.py" --mode temp --active-window
python3 "$SS_SKILL/scripts/take_screenshot.py" --mode temp --app "Google Chrome"
mkdir -p output/screenshots
python3 "$SS_SKILL/scripts/take_screenshot.py" --path output/screenshots/final-validation.png

Acceptance criteria

  • Screenshots exist for all required validation states.
  • No obvious visual regressions or missing UI elements.
  • Artifacts are saved for instructor review.

Exercise 10: Feature D - Multilingual UI (English + Spanish)

10-20 minutes Localization

Goal: make the UI multilingual with runtime language switching and persistence. Optional follow-on exercise for this phase.

Codex setup (App or CLI)

  • Codex App: Reuse your repo session in Codex App and run: Add multilingual support (English and Spanish): language selector, translation dictionary, UI label wiring, and tests for language toggle behavior.
  • Codex CLI: Run codex in repo root (or reuse your open session), then run the same prompt.

Build

  • Add a language selector supporting en and es.
  • Move user-facing strings into a translation map/object.
  • Persist selected language and use English fallback behavior.

Tasks

  1. Identify all visible labels in dashboard, filters, and table headers.
  2. Implement translation mapping and selector wiring with Codex.
  3. Apply translations across key UI surfaces.
  4. Add/update tests for toggle behavior and fallback.

Acceptance criteria

  • Users can switch between English and Spanish at runtime.
  • Key labels update without a full page reload.
  • Selected language persists after refresh.
  • Translation-related tests pass.
